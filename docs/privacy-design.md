# shh — Privacy Design (cryptographic source of truth)

This document fixes every hash, field element, and circuit signal. Circuits, contracts,
and the SDK **must** agree with it. If you change something here, change all three and
re-run the trusted setup.

## Primitives

- **Field**: BN254 scalar field `p = 21888242871839275222246405745257275088548364400416034343698204186575808495617`.
  All commitments, nullifiers, and Merkle nodes are elements of `F_p`.
- **Hash**: Poseidon (circomlib parameters). Notation `H(a, b, …)` = Poseidon over those inputs.
  On-chain Poseidon is the EVM contract generated by `circomlibjs` `poseidonContract`,
  matching the circuit's Poseidon exactly.
- **Merkle tree**: fixed depth `LEVELS = 20` (1,048,576 leaves), binary, Poseidon(2) per node.
  Empty subtree value `ZERO = keccak256("shh") mod p` (a nothing-up-my-sleeve zero leaf).
  Node order: `parent = H(left, right)` where `left/right` are chosen by the path index bit.

## Profile B — fixed-denomination Privacy Pool

A pool is parameterised by a single `denomination` (e.g. 0.1 ETH). Every deposit is for
exactly that amount.

### Secrets & public values

| Name            | Type                   | Where                                 |
| --------------- | ---------------------- | ------------------------------------- |
| `nullifier`     | random `F_p`           | secret                                |
| `secret`        | random `F_p`           | secret                                |
| `commitment`    | `H(nullifier, secret)` | public on deposit, hidden on withdraw |
| `nullifierHash` | `H(nullifier)`         | public on withdraw                    |

### Trees

- **State tree** (`stateRoot`): every deposited `commitment`, in insertion order.
- **Association tree** (`associationRoot`): the subset of commitments the ASP marks as
  compliant. Same leaf value (`commitment`) so one secret proves membership in both.

### Withdraw circuit — `pool/poolWithdraw.circom`

Public signals (in order):

```
[ stateRoot, associationRoot, nullifierHash, recipient, relayer, fee, refund ]
```

Private inputs:

```
nullifier, secret,
statePathElements[LEVELS],  statePathIndices,
assocPathElements[LEVELS],  assocPathIndices
```

Constraints:

1. `commitment = H(nullifier, secret)`.
2. `nullifierHash === H(nullifier)`.
3. MerkleProof(commitment, statePathElements, statePathIndices) === `stateRoot`.
4. MerkleProof(commitment, assocPathElements, assocPathIndices) === `associationRoot`.
5. `recipient, relayer, fee, refund` are bound (squared) so the proof can't be reused with
   different withdrawal parameters (anti-front-running).

The contract checks `nullifierHash` is unused, marks it spent, verifies the proof against the
**current** `stateRoot` (or a recent historical root) and the ASP's published `associationRoot`,
then pays `denomination − fee` to `recipient` and `fee` to `relayer`.

> **Unlockable / compliance.** Because withdrawal _requires_ membership in the association
> tree, the ASP governs which deposits can ever exit privately. Excluding a tainted deposit
> from `associationRoot` leaves it deposited but non-withdrawable through the private path —
> this is the regulatory "equilibrium" of Privacy Pools.

## Profile A — shielded UTXO pool

UTXO model (after Tornado Nova): arbitrary amounts, join-split transfers.

### Note & keys

| Name              | Definition                                    |
| ----------------- | --------------------------------------------- |
| `privKey`         | random `F_p`                                  |
| `pubKey`          | `H(privKey)`                                  |
| note `commitment` | `H(amount, pubKey, blinding)`                 |
| `signature`       | `H(privKey, commitment, merklePathIndices)`   |
| `nullifier`       | `H(commitment, merklePathIndices, signature)` |

`blinding` is a random `F_p` per note. `amount` is constrained to 248 bits to prevent
field overflow during summation.

### Transaction circuit — `shielded/transaction.circom` (instantiated 2-in / 2-out)

Public signals (in order):

```
[ root, publicAmount, extDataHash,
  inputNullifier[nIns], outputCommitment[nOuts] ]
```

Private inputs: per-input `inAmount, inPrivateKey, inBlinding, inPathIndices,
inPathElements[LEVELS]`; per-output `outAmount, outPubkey, outBlinding`.

Constraints:

1. For each input: recompute `commitment`, `signature`, `nullifier`; `nullifier === inputNullifier[i]`.
2. For each input with non-zero amount: MerkleProof(commitment, …) === `root`
   (zero-amount inputs are dummies and skip the root check).
3. For each output: recompute `commitment`; `=== outputCommitment[i]`; `amount` fits in 248 bits.
4. All input nullifiers are pairwise distinct.
5. **Value conservation**: `Σ inAmount + publicAmount === Σ outAmount` (mod p).
6. `extDataHash` is bound (squared) — it commits to recipient, relayer, fee, and the
   encrypted output ciphertexts the contract emits for note discovery.

`publicAmount` is computed by the contract as `(extAmount − fee) mod p`:

- `extAmount > 0` → deposit (caller sends `extAmount`),
- `extAmount < 0` → withdrawal (contract pays `−extAmount` to recipient),
- `extAmount = 0` → internal private transfer.

### Spending & double-spend protection

On `transact`, the contract: verifies the proof against a known `root`, checks each
`inputNullifier` is unused and marks it spent, inserts each `outputCommitment` into the tree,
emits encrypted outputs, and settles `publicAmount` against the bridge/escrow.

## Shielded bridge (Base ↔ shh)

Locking `denomination`/amount on Base emits a message via the OP Stack messenger; on L3 the
`ShieldedBridge` inserts the resulting commitment directly into the shielded set, so bridged
funds arrive as a spendable note rather than a transparent balance. Withdrawal reverses it: a
shielded burn proof on L3 authorizes an unlock on Base.

## Cross-component invariant checklist

- [ ] Poseidon arity and constants identical in circom and the deployed hasher.
- [ ] `LEVELS`, `ZERO`, and node ordering identical in circuit, `MerkleTreeWithHistory`, SDK.
- [ ] Public-signal ordering identical in circuit, exported verifier, and contract call sites.
- [ ] `amount` bit-width (248) enforced in circuit and respected by the SDK.
